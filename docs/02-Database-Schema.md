# 2. Database Schema

The platform uses MongoDB with Mongoose for Object Data Modeling. The schema is highly normalized to separate concerns between authentication, clinical profiles, and session data.

## Core Models

### `User`
The central authentication entity. Stores email, hashed password (or OAuth provider IDs), and the fundamental `role` (`client`, `psychologist`, `psychiatrist`, `admin`). Practitioner-specific fields (like `slmcRegNo`) are embedded here for quick validation.

### `ClientProfile`
Stores the client's medical history, emergency contacts, and therapist assignments. This sits separate from the `User` model to strictly separate authentication data from protected health information (PHI).

### `Session`
Represents an appointment between a client and a practitioner. Tracks `status` (pending, confirmed, completed, cancelled) and `type` (online, physical).

## Clinical Models

### `SessionNote`
Strictly private notes written by the practitioner during or after a session. These are never exposed to the `client` role.

### `MoodEntry`
Idempotent, timestamped daily entries created by the client (rating 1-10 plus journaling). Aggregated on the client dashboard to render progress charts.

### `Goal`
Actionable items assigned by the practitioner for the client to complete between sessions. Replaces verbal homework with trackable metrics.

### `Prescription`
Stores base64 strings of the psychiatrist's signature and official seal, along with medication details. Designed for zero-persistence PDF generation (the PDF is generated dynamically and never stored on disk).

## Communication Models

### `Message`
Instant messages sent between client and practitioner. Always includes a `flagged` boolean that is set by the active Keyword Safety Engine.

### `KeywordAlert`
A notification document generated by the server when a `Message` trips a crisis keyword. Links to the originating message and tracks the action status (pending/acknowledged by practitioner).
